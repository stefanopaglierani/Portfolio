Server-Side Validation & State Management in Procedural Environments

By Stefano Paglierani • Est. Read: 10 mins • Systems Architecture

In developing complex simulation environments within Roblox Studio, maintaining the integrity of the client-server boundary is paramount. When building a procedural survival game with an intricate crafting and building system, allowing the client to dictate state changes is a fundamental security flaw that invites exploits and synchronization errors.

This technical dive explores the architecture of a custom BuilderService, a robust server-side module designed to handle crafting logic, resource validation, and procedural asset instantiation securely.

The Architecture Problem: Secure Crafting Logic

The core challenge is processing dynamic player requests (e.g., "Build a Moisture Extractor") against a complex set of dependencies—checking shared global resources, verifying propaedeutic technology unlocks, and ensuring the correct procedural placement of the asset in the world—all without halting the server thread or creating execution bottlenecks.

1. The Data Structure: The Recipe Dictionary

To ensure scalability, the crafting logic is decoupled from the execution logic[cite: 1]. I implemented a centralized, dictionary-based data structure (CRAFT_RECIPES) that acts as the single source of truth for all buildable assets[cite: 1].

This approach allows for rapid balancing and the addition of new items without modifying the core validation algorithms[cite: 1]. Each entry defines resource costs, item types (Structure, Upgrade, VIP), and propaedeutic requirements (Requires)[cite: 1].

-- 📖 IL LIBRO DELLE RICETTE
local CRAFT_RECIPES = {
    ["MoistureExtractor"] = { Rock = 15, Permafrost = 5, Type = "Structure", Requires = nil },
    ["PlanterBox"] = { Rock = 10, Permafrost = 5, Type = "Structure", Requires = "MoistureExtractor" },
    ["ResidentialCoreSystem"] = { Rock = 50, Crystal = 5, Gold = 20, Type = "Expansion", Slot = "Slot_Residential", Blueprint = "Residential" },
    ["TechDrillLV2"] = { Rock = 20, Gold = 5, Type = "Upgrade", Requires = nil },
    ["MechaSuite"] = { Type = "VIP", GamepassID = 987654321 }      
}

Validation & Procedural Instantiation

When a client issues a BuildRequest via a RemoteEvent, the server intercepts the call and runs it through a strict, multi-tiered validation pipeline[cite: 1].

2. The Validation Pipeline

The server independently verifies the request against three critical parameters before any resources are deducted or assets spawned[cite: 1]:

  1. Premium Access Security: If the requested item is tagged as "VIP", the script queries the MarketplaceService to asynchronously verify ownership, logging potential bypass attempts[cite: 1].
  2. Propaedeutic Checks: It verifies if the player holds the required custom attributes (e.g., verifying if a player has built a MoistureExtractor before allowing a PlanterBox)[cite: 1].
  3. Resource Verification: A dedicated function (checkResources) compares the recipe costs against real-time global values stored in ReplicatedStorage[cite: 1].
-- 🛑 Controllo Requisiti Base (Propedeuticità)
if recipe.Requires and not player:GetAttribute("Has_" .. recipe.Requires) then
    aresResponse:FireClient(player, "ACCESS DENIED: BUILD " .. string.upper(recipe.Requires) .. " FIRST.", true)
    return
end

-- 💰 Controllo Risorse Multiplo
local function checkResources(recipe)
    if recipe.Rock and storedRock.Value < recipe.Rock then return false end
    if recipe.Permafrost and storedPermafrost.Value < recipe.Permafrost then return false end
    if recipe.Gold and storedGold.Value < recipe.Gold then return false end
    if recipe.Crystal and storedCrystal.Value < recipe.Crystal then return false end
    return true
end

3. Dynamic Asset Spawning

Once validated, the system handles the physical manifestation of the request[cite: 1]. The spawnStructure function handles the procedural placement of models[cite: 1]. It locates a designated spawn folder in the Workspace, selects a random available location node (to prevent structural overlap), calculates the correct bounding box offset to ensure the model sits flush with the terrain, and pivots the cloned asset into position[cite: 1]. The location node is then destroyed to prevent future overlapping spawns[cite: 1].

local function spawnStructure(itemName, player)
    local templateName = (itemName == "MoistureExtractor") and "watertower" or itemName
    local template = ReplicatedStorage:FindFirstChild(templateName)
    local locations = Workspace:FindFirstChild(itemName .. "Locations")

    -- Validazione robusta dei target
    if not template then return false, "SYSTEM ERROR" end
    if not locations or #locations:GetChildren() == 0 then return false, "NO BUILD LOCATION" end

    local spots = locations:GetChildren()
    local chosen = spots[math.random(1, #spots)]
    local clone = template:Clone()

    -- Calcolo dell'offset in base alla classe dell'oggetto
    local halfHeight = 0
    if clone:IsA("Model") then
        local _, size = clone:GetBoundingBox()
        halfHeight = size.Y / 2
    elseif clone:IsA("BasePart") then
        halfHeight = clone.Size.Y / 2
    end

    local finalCFrame = chosen.CFrame * CFrame.new(0, halfHeight, 0)
    if clone:IsA("Model") then clone:PivotTo(finalCFrame) else clone.CFrame = finalCFrame end

    clone.Parent = Workspace
    chosen:Destroy()
    return true, ""
end

Dynamic Inventory Management

Beyond structural spawns, the script dynamically manages player inventory upgrades[cite: 1]. When an upgrade like a TechDrillLV2 or a BackPackLV3 is validated, the server iterates through both the player's Backpack and physically equipped Character to locate and destroy obsolete instances of the tool before cloning and injecting the upgraded version[cite: 1]. This ensures inventory state consistency and prevents item duplication exploits[cite: 1].

Conclusion

By enforcing a strict validation pipeline on the server and utilizing dictionary-driven logic, the BuilderService ensures that the simulation remains secure, scalable, and modular[cite: 1]. It serves as a prime example of translating complex operational logic into a robust backend architecture.
